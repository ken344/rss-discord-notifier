# Architecture Decision Records (ADR)

このディレクトリには、RSS Discord Notifierプロジェクトにおける重要なアーキテクチャ決定を記録したADR（Architecture Decision Records）が含まれています。

## ADRとは？

ADRは、ソフトウェア開発における重要な設計決定を記録するためのドキュメントです。各ADRは以下の情報を含みます：

- **コンテキスト**: なぜこの決定が必要だったのか
- **検討した選択肢**: どのような代替案があったか
- **決定**: 最終的に何を選択したか
- **理由**: なぜその選択をしたのか
- **影響**: その決定がプロジェクトにどのような影響を与えるか

### [ADR-0001: Go言語の採用](0001-use-go-language.md)
**ステータス**: 承認済み  
**概要**: プロジェクトの実装言語としてGo言語を選択した理由と、他の選択肢との比較。

**主な決定要因**:
- シングルバイナリ配布の容易さ
- 並行処理の実装しやすさ
- GitHub Actionsとの親和性
- 標準ライブラリの充実

**ステータス**: 承認済み

## ADRの命名規則

ADRファイルは以下の命名規則に従います：

```
NNNN-short-title.md
```

- `NNNN`: 4桁の連番（0001から開始）
- `short-title`: 決定内容を表す短いタイトル（ケバブケース）
- `.md`: Markdown形式

例: `0001-use-go-language.md`

## ADRのテンプレート

新しいADRを作成する際は、以下のテンプレートを使用してください：

```markdown
# ADR NNNN: [決定のタイトル]

## ステータス
[提案中 | 承認済み | 却下 | 廃止 | 置き換え済み]

## コンテキスト
[決定が必要になった背景や状況を説明]

### 要求事項
- [要求事項1]
- [要求事項2]

### 検討した選択肢
#### 1. [選択肢1]
**概要**: [説明]

**メリット**:
- [メリット1]

**デメリット**:
- [デメリット1]

#### 2. [選択肢2]
...

## 決定
[何を決定したか]

## 理由
[なぜその決定をしたか]

## 影響
### ポジティブ
- ✅ [良い影響1]

### ネガティブ
- ⚠️ [課題1]

### リスク軽減策
- [対策1]

## 関連決定
- [関連するADRへのリンク]

## 参考
- [参考資料へのリンク]
```

## ADRの更新方針

- **承認済みのADRは原則として変更しない**: 決定の履歴を保持するため
- **決定が変更された場合**: 新しいADRを作成し、古いADRのステータスを「置き換え済み」に更新
- **誤字脱字の修正**: 内容を変えない範囲での修正は可能
